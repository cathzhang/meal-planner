# 需求总览

> 本文档汇总本项目从设计到实现的所有需求，用于回顾和后续迭代参考。

---

## 一、项目定位

**家庭菜单推荐系统**。核心目标：根据家庭成员、季节、时间、预算等约束，智能推荐"今天吃什么"，并生成购物清单。

---

## 二、菜品数据结构（Dish）

每道菜需要记录的字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `name` | str | 菜名 |
| `ingredients` | List[str] | **主要食材**（普通油盐等基础调料不列入） |
| `sauces` | List[str] | **特殊酱料**（豆瓣酱、蚝油等需要单独采购的） |
| `cook_time_minutes` | int | 灶上实际制作时间 |
| `prep_time_minutes` | int | **备菜时间**（切配、腌制等） |
| `difficulty` | Enum | 简单 / 中等 / 困难 |
| `dish_type` | Enum | **大荤** / **小荤** / **全素** |
| `summer_recommended` | bool | 是否夏日推荐 |
| `winter_recommended` | bool | 是否冬日推荐 |
| `spicy_level` | Enum | **不辣** / **可免辣** / **辣**（三选一） |
| `kid_friendly` | bool | **是否儿童友好** |
| `seasonal_months` | List[int] | 适合月份 `[1-12]`，**空列表=全年都有** |
| `price_level` | Enum | **便宜** / **中等** / **较贵** |
| `has_prepackaged` | bool | 是否有成熟预制菜（方便买菜时对比） |
| `variant_group` | Optional[str] | **变体组**（如"排骨汤系列"），同组菜可互相替换 |
| `liked_by` | List[str] | 家庭成员偏好名单 |
| `is_soup` | bool | **是否是汤菜** |

### 字段设计演进

1. **初始版本**：食材、酱料、制作时间、难度、类型（大荤/小荤/全素）、季节（夏/冬布尔值）、喜好人员
2. **辣味合并**：`spicy` + `spicy_removable` 两个布尔值 → 合并为 `spicy_level` 三选一枚举
3. **时间拆分**：`time_minutes` → 拆分为 `cook_time_minutes` + `prep_time_minutes`
4. **季节细化**：布尔值 → `seasonal_months` 月份列表（支持跨季节食材如莲藕秋季、荠菜春季）
5. **新增维度**：儿童友好、菜价评估、预制菜标识、变体组、汤菜标识

## 三、用餐场景配置（MealConfig）

推荐前需要用户确定的场景参数：

| 参数 | 类型 | 说明 |
|---|---|---|
| `people_count` | int | **吃饭人数** |
| `day_type` | str | **"工作日"** 或 **"周末"** |
| `need_soup` | bool | 是否需要汤 |
| `target_main_dishes` | Optional[int] | 主菜数量，None=按人数自动计算 |
| `existing_dishes` | List[str] | **已有菜品**（剩菜/打包菜），会从推荐中扣除 |

### 核心逻辑

- **默认配餐公式**：n 个人 → n-1 个主菜 + 1 个汤（可配置）
- **工作日筛选**：只推总耗时 ≤35 分钟、难度不高于"中等"的菜
- **周末放开**：可以推荐费工夫的大菜（红烧、炖煮类）

## 四、数据存储方案

采用**双层存储**：

```
data/recipes.json   ← 核心数据（文本，git 追踪，人类可读）
        ↓
  python import_data.py
        ↓
meal_planner.db     ← SQLite 数据库（运行时生成，.gitignore 忽略）
```

### 工具链

| 脚本 | 功能 |
|---|---|
| `add_recipe.py` | **交互式一问一答**录入新菜品，自动追加到 JSON |
| `import_data.py` | 从 JSON **重建数据库** |
| `cli.py` | 命令行 **CRUD**：list / show / search / add / edit / delete / stats |

### 核心数据安全

- `data/recipes.json` 是**唯一真相来源**，必须加入 git
- `meal_planner.db` 是二进制文件，不加入 git，丢失可随时重建

## 五、推荐算法设计

### 不适用传统算法的原因

- 没有用户评分矩阵 → **协同过滤/矩阵分解不可用**
- 推荐目标是"一餐的组合"而非"单个商品"
- 约束条件多且硬（荤素、季节、时间、儿童、辣味）

### 推荐方案（三层架构）

| 层级 | 模型 | 负责什么 |
|---|---|---|
| **第一层** | **多因素加权评分** | 给每道菜按规则打分（季节+20、家人爱吃+30、近期没做+50…），过滤明显不合适的 |
| **第二层** | **组合优化**（穷举/贪心） | 在一餐框架内选最优搭配（如大荤+小荤+素+汤），考虑口味不重复、时间可并行 |
| **第三层** | **基于内容的相似推荐** | 辅助功能：缺食材时推荐同组变体、口味互补搭配建议 |
| **第四层**（预留） | **关联规则挖掘**（Apriori） | 数据积累足够后叠加：挖掘"常一起做的菜" |

### 数学基础

- **加权评分**：线性加权求和
- **组合优化**：小规模穷举（家庭菜谱通常几十道，组合数可控）
- **相似度**：Jaccard 相似度（食材重叠度）

## 六、后续可扩展方向

1. **推荐算法实现**：把三层架构落地为可运行的推荐函数
2. **历史记录**：记录每天的菜单，用于"近期不重复"和长期关联规则挖掘
3. **购物清单**：推荐结果 → 自动合并食材和酱料 → 生成采购清单
4. **Web 界面**：用 datasette 或 Flask 做可视化浏览
5. **手机端**：后续可考虑小程序或 Web App
